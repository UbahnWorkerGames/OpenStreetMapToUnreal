UEMap → PCG Graph-Traversal: Machbarkeitsanalyse

Basierend auf Stand v0.1.37 (main.js, 6425 LOC) — 30. Mai 2026

⚡ TL;DR

Ja, prinzipiell machbar. Aber der einfachste Hebel ist NICHT das Graph-Traversal, sondern alle Splines in EIN PCG-Volume-Setup zu werfen. Das ist mit dem existierenden buildAreaPcgSplines()-Export schon 80% da — du brauchst nur das UE-seitige PCG-Graph-Setup. Der "Startpunkt → durchlaufen" Ansatz ist Overkill und löst ein Problem das du gar nicht hast.

Merge/Stitch/Collapse Pipeline

  1. mergeAreaCenterlines(): Gruppiert Features nach category + name (z.B. "city_road_hauptstrasse") und merged parallele Richtungsfahrbahnen zu einer Centerline — nur wenn sie nahe genug beieinander liegen (Road: ≤45m, Sonstige: ≤18m).
  2. stitchConnectedAreaFeatures(): Verbindet Features derselben Kategorie+Name deren Endpunkte ≤35m (Export) auseinanderliegen. Iterativ bis keine Stitches mehr möglich.
  3. collapseAreaRoadGroupToAxis(): Für Richtungsstraßen: ALLE Feature-Punkte einer Named-Group werden auf eine einzige Achse projiziert (PCA-ähnlich via roadAxisVector). Aus 2+ Richtungsfahrbahnen wird 1 Centerline.

Aktuelle Schwachstelle: Kein echter Graph!

stitchConnectedAreaFeatures() verbindet nur Features derselben Kategorie UND desselben Namens. Das heißt:

  • Autobahn (motorway) wird NIE mit Stadtstraße (city_road) verbunden
  • Service-Road ohne name wird NIE mit anderer Service-Road verbunden
  • Kreuzungen zwischen verschiedenen Straßentypen existieren NICHT im Datenmodell

Du hast kein Straßennetz-Graph — du hast 1500 isolierte Spline-Stücke.

2. Deine Idee: Startpunkt → PCG traversiert das Straßennetz

Was du beschreibst

Statt 1500 einzelner Spline-Actors in UE: Ein Startpunkt auf einer Autobahn, und PCG "wächst" von dort aus alle verbundenen Straßen als ein zusammenhängendes Konstrukt. Das Ergebnis ist 1 Actor mit einem Graph-basierten Straßennetz statt 1500 Einzel-Actors.

2.1. Was dafür nötig wäre — Graph-Extraktion (UEMap-Seite)

Intersection Detection

Du musst ALLE Kreuzungspunkte zwischen ALLEN Straßen-Features finden, unabhängig von Kategorie/Name:

// Pseudocode für Graph-Bau
function buildRoadGraph(features) {
  const graph = { nodes: new Map(), edges: [] };
  
  // Schritt 1: Alle Endpunkte sammeln
  for (const feature of features) {
    const start = feature.controlGeometry[0];
    const end = feature.controlGeometry.at(-1);
    
    // Schritt 2: Endpunkte mit Toleranz matchen (z.B. 10m)
    const startNode = findOrCreateNode(graph, start, 10);
    const endNode = findOrCreateNode(graph, end, 10);
    
    // Schritt 3: Kante einfügen
    graph.edges.push({
      from: startNode, to: endNode,
      category: feature.category,  // motorway, city_road, ...
      name: feature.name,
      geometry: feature.controlGeometry,  // Spline-Punkte
      osmClass: feature.tags?.highway,
      width: areaFeatureWidthM(feature),
      oneway: feature.tags?.oneway === "yes",
      // ... weitere Attribute
    });
  }
  
  // Schritt 4: Interior-Intersections finden
  // Zwei Straßen können sich irgendwo in der Mitte kreuzen,
  // nicht nur an den Endpunkten!
  for (let i = 0; i < edges.length; i++) {
    for (let j = i+1; j < edges.length; j++) {
      const intersection = segmentIntersection(edges[i], edges[j]);
      if (intersection) {
        // Edge splitten, neuen Node an Kreuzung erstellen
        splitEdgeAtIntersection(graph, i, j, intersection);
      }
    }
  }
  
  return graph;
}

Das ist ein nicht-triviales Geometrie-Problem. Segment-Intersection-Tests für 1500 Features sind quadratisch (~1.1M Paare) — brauchst Spatial Index (R-Tree, Quadtree).

Realistische Probleme

  • Unconnected Subgraphs: Viele Straßen enden im Nichts (Sackgassen, Feldwege ohne Verbindung). Dein "von einem Startpunkt aus alles traversieren"-Ansatz erreicht nur die Zusammenhangskomponente des Startknotens — 30-50% der Straßen bleiben unerreicht.
  • Tolerance-Handling: Welcher Abstand zählt als "verbunden"? 5m? 15m? Verschiedene Straßentypen haben verschiedene physische Kreuzungs-Geometrien. Falsche Toleranz → falsche Verbindungen oder fehlende Verbindungen.
  • Overpass-Datenqualität: OSM-Ways sind an Kreuzungen oft nicht exakt verbunden. Zwei Ways die sich im echten Leben kreuzen, teilen in OSM nicht immer einen Node. Du musst geometrisch mergen.
  • Brücken/Tunnel: Eine Autobahn kreuzt eine Stadtstraße — physisch verbunden (Brücke) oder nicht? tags.layer, tags.bridge, tags.tunnel müssen berücksichtigt werden.

2.2. Was dafür nötig wäre — PCG-Seite (Unreal Engine)

Varianten für PCG-Integration

Variante A: "Dumme" Spline-Liste in PCG (80% Lösung, existiert fast)

Exportiere ALLE Splines als PCG DataTable (was buildAreaPcgSplines() heute schon tut). Ein PCG Volume + Spline Sampler + Mesh Spawner generiert alle Straßen-Meshes. Kein Graph-Traversal nötig. Vorteil: 1 PCG Actor statt 1500 BP Actors.

Variante B: Graph-JSON + Custom PCG Element (100% Lösung, viel Arbeit)

Exportiere den Graph als JSON (Nodes + Edges). In UE: Custom PCG Element (C++ oder Blueprint) das den Graph traversiert und Straßen-Meshes generiert. Ermöglicht echtes "von Punkt A nach B das Netz aufspannen".

Variante C: Hybrid — Splines mit Connectivity-Metadaten (90% Lösung)

Jeder Spline bekommt im Export zusätzliche Felder: ConnectedTo: ["spline_key_1", "spline_key_2"]. PCG kann dann an Kreuzungen saubere Übergänge spawnen ohne den ganzen Graph traversieren zu müssen.

3. Ehrliche, knallharte Meinung

⚠️ Technisch machbar, aber: Du optimierst das falsche Problem.

Was diese Idee LÖST

  • ✅ Weniger Actors in UE (1500 → 1 PCG Volume)
  • ✅ Sauberere Kreuzungen (PCG kann Übergänge zwischen Straßen meshen)
  • ✅ Bessere Performance im Editor (Outliner hat nicht 1500 Einträge)

Was diese Idee NICHT löst

  • ❌ Straßen sind immer noch diskrete Segmente — nur halt in einem Graph statt in 1500 Actors
  • ❌ Performance zur Laufzeit: PCG muss Geometrie generieren, ist nicht unbedingt schneller
  • ❌ Der Graph-Traversal-Aspekt ("von Startpunkt aus wachsen") bringt keinen Mehrwert wenn eh alle Straßen im Volume sind

Das EIGENTLICHE Problem

Dein Problem ist nicht "5000 Actors sind zu viele". Dein Problem ist: Jeder Actor ist ein dünner Spline, und du willst daraus realistische Straßen mit Breite, Markierungen, Bürgersteigen, Kreuzungen machen.

Das löst PCG — aber du brauchst NICHT den Graph-Traversal-Teil dafür. PCG mit Spline-Daten als Input reicht völlig. Die Frage ist nur: Wie gut kann dein PCG-Graph die Straßengeometrie aus den Spline-Daten generieren?

4. Konkrete Umsetzungs-Empfehlung

Phase 1: PCG "Dumb Mode" (2-3 Tage Aufwand)

  1. Export verbessern: buildAreaPcgSplines() liefert heute schon Spline-Punkte als DataTable-Zeilen. Füge pro Zeile hinzu: SplineLength, SplineWidth, RoadType, IsBridge, IsTunnel.
  2. UE PCG Graph bauen:
    • PCG Volume → Get Spline Data (from DataTable) → Spline Sampler → Mesh Spawner
    • Pro RoadType unterschiedliche Meshes (Autobahn breit + Leitplanke, Stadtstraße schmal + Bordstein)
    • An Kreuzungen: Spline-Endpunkte mergen (Point Cloud → Density Filter → Static Mesh Spawner für Kreuzungs-Plates)
  3. Testen: Funktioniert das Mesh-Generieren aus Spline-Daten? Wie sieht die Qualität aus?

Das ist der schnelle Win. Wenn das schon gut aussieht, brauchst du Phase 2+3 gar nicht.

Phase 2: Intersection Detection (3-5 Tage)

  1. R-Tree/Spatial-Index in main.js für schnelle Segment-Intersection-Tests
  2. Alle Straßen-Spline-Paare auf Kreuzungen prüfen
  3. An Kreuzungen: Splines splitten, Node-IDs vergeben, Connectivity-Metadaten speichern
  4. Export-Format: { nodes: [...], edges: [...] } statt flacher Spline-Liste

Phase 3: Custom PCG Graph Element (5-10 Tage)

  1. C++ PCG Element das den Graph-JSON einliest
  2. Traversal-Logik: Von Seed-Node(s) aus BFS über den Graph
  3. Pro Edge: Spline generieren, Mesh spawnen, an Nodes: Kreuzungs-Meshes
  4. Road-Hierarchy: Motorway-Kanten priorisieren, nur verbundene subgraphs rendern

Das ist der teure Teil. Erst machen wenn Phase 1+2 nicht reichen.

5. Kann man das mit Coding Agents bauen?

✅ JA — aber nicht am Stück. Iterativ, Phase für Phase.

Was Coding Agents GUT können (für dieses Projekt)

  • Phase 1 (PCG "Dumb Mode"): Perfekt. DataTable-Format erweitern, UE PCG Graph per Python/PythonScript bauen. Das ist stupide Fleißarbeit — genau was Agents gut machen.
  • Phase 2 (Intersection Detection): Agents können den R-Tree einbauen, Segment-Intersection-Mathe implementieren. ABER: Du musst die Toleranz-Parameter selber kalibrieren (5m? 15m?). Ein Agent rät nur.
  • Phase 3 (Custom PCG Element): Agents können C++ Boilerplate generieren, aber das Debuggen im UE-Editor ist HANDARBEIT. Kein Agent kann den UE-Editor bedienen oder visuell prüfen ob die Kreuzungen gut aussehen.

Was Coding Agents NICHT können

  • Visuelle Qualitätskontrolle: "Sieht die Kreuzung gut aus?" — kann kein Agent beantworten. Du musst in UE reinschauen.
  • UE-spezifisches Debugging: PCG Graph crasht, Splines sind kaputt, Mesh sieht falsch aus — der Agent kann nicht den UE-Editor inspizieren.
  • Parameter-Tuning: Toleranzen, Road-Widths, Mesh-Offsets — trial and error. Agent kann Werte vorschlagen aber nicht testen.
  • Komplexe Geometrie-Debugging: Wenn die Intersection-Detection bei einer bestimmten Straßenkonfiguration falsch liegt, findest du das nur durch visuelle Inspektion im UEMap-UI.

Recommended Agent-Workflow

  1. Ein Agent pro Phase, nicht ein Monster-Agent für alles.
  2. Agent macht den Code → du testest visuell → Agent fixt basierend auf deinem Feedback.
  3. Jede Iteration committen (machst du ja schon mit Git-First). So kannst du "Scheiße, das war vorher besser" immer zurückspielen.
  4. Phase 1 zuerst fertigstellen und deployen bevor du Phase 2 anfängst. Wenn Phase 1 schon "gut genug" ist — hör auf.

🏆 Meine Empfehlung

Bau Phase 1 (PCG Dumb Mode) zuerst. Du hast 80% des Codes schon. Exportiere die Splines als PCG DataTable, bau einen PCG Graph in UE der daraus Straßen mesht. Wenn das Ergebnis scheiße aussieht, brauchst du Phase 2+3. Wenn es gut aussieht — Glückwunsch, du hast in 2-3 Tagen erreicht was du wolltest, ohne den Graph-Traversal-Irrsinn.

Der "Startpunkt → Straßen wachsen" Teil der Idee ist cool aber overengineered. PCG verarbeitet von Natur aus ALLES im Volume — du brauchst kein Traversal um "alles zu erreichen". Das einzige Szenario wo Traversal Sinn macht: Du willst nur einen Teil des Netzes rendern (z.B. "zeig mir alle Straßen die von dieser Autobahn aus erreichbar sind"). Aber willst du das wirklich?
